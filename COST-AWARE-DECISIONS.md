# Cost-Aware Decisions

This document records decisions, assumptions, trade-offs, evidence, affected files or services, validation findings, and unresolved issues from cost-aware scalability and cost-efficiency work.

Use this structure where applicable:

| Field | Content |
| --- | --- |
| What changed | The specific planned or implemented modification. |
| Where | Files, services, configs, tests, docs, or runtime components affected. |
| Why | The reason for the change. |
| Cost driver targeted | The resource, operational cost, or waste pattern addressed. |
| Evidence | Repository evidence, workload assumption, metric, test result, or source document. |

## Cost-Aware Scalability Prompt Set

### 2026-06-14 - Cost-Aware Docker Compose Scaling Plan

The following entries document planned changes only. No implementation code or Compose changes were generated in this step.

| Planned change | Where | Why proposed | Cost driver targeted | Evidence | Expected cost impact | Risk or trade-off | Validation method | Status |
| --- | --- | --- | --- | --- | --- | --- | --- | --- |
| Keep Docker Compose as the only deployment mechanism and express scale through Compose overrides/profiles plus `docker compose up --scale`. | `docker-compose.yml`, future Compose overrides, `SCALABILITY.md` | The source documents require Docker Compose and a shared named Docker network; the user explicitly excluded Kubernetes, Helm, Nomad, Swarm, Terraform, managed cloud services, and other orchestrators. | Avoids operational/provisioning complexity and extra control-plane cost. | `ARCHITECTURE.md` benchmarking context; `REQUIREMENTS.md` M-18, M-19, M-20; `TECH-STACK.md` Docker Compose runtime choice. | Keeps the benchmark cheaper and easier to reproduce locally. | Docker Compose config validation for each profile and service inventory checks. | Planned |
| Use a Docker Compose gateway/reverse proxy as the single benchmark entrypoint. | Gateway service, Nginx config, k6 base URL, Prometheus/Grafana gateway metrics. | Horizontal service replicas need a stable entrypoint so k6 does not target individual containers. | Avoids duplicated test scripts and unnecessary per-service exposed ports; reduces routing complexity. | Existing `docker-compose.yml` includes a gateway-facing runtime in current validation history; M-19 requires Compose-started infrastructure; M-21 requires load generation of main flows. | A tuned single gateway can serve smoke/calibration cheaply before adding gateway replicas. | Gateway `nginx -t`, `/health`, Nginx exporter metrics, k6 smoke through gateway. | Planned |
| Scale Auth Service modestly to 3-4 replicas for full benchmark profiles. | Auth service Compose scale profile and auth-db pool settings. | Auth handles about 500 logins/s, but protected services validate JWTs locally and do not call Auth per request. | Avoids over-provisioning a service that is not in the per-request critical path. | Workload table: ~500 auth logins/s; `ARCHITECTURE.md` and `REQUIREMENTS.md` M-03 require local JWT validation by services. | Lower CPU/memory spend than scaling Auth with every protected request. | k6 auth-login scenario, auth-db connection pool metrics, login p95/p99 latency. | Planned |
| Scale Catalog Service to 4-6 replicas only after pagination and database indexes are confirmed. | Catalog service Compose scale profile, catalog-db indexes, catalog pagination config. | Catalog is read-heavy and participates in the 4,000 catalog/search requests/s path. | DB CPU, app CPU, connection count, repeated page scans. | Workload table: ~4,000 catalog/search requests/s; `REQUIREMENTS.md` M-07, M-08, M-09 require paginated browsing, detail lookup, startup ingestion. | App replicas are cheaper than DB scaling only while catalog-db is not saturated. | k6 catalog scenario, slow query review, datasource pool metrics, catalog p95 latency. | Planned |
| Prevent duplicate Catalog startup ingestion during scaled runs by planning a single ingestion-owner mode or equivalent controlled startup. | Catalog service startup configuration, future Compose profile/env var. | Scaling Catalog replicas can cause repeated dataset ingestion work unless one replica owns seeding. | Startup CPU, database write I/O, duplicate work. | `REQUIREMENTS.md` M-09 requires automatic dataset ingestion; current Compose exposes `CATALOG_INGESTION_ENABLED`. | Reduces startup cost and DB pressure in scaled profiles. | Scaled startup smoke, catalog row-count stability, ingestion logs showing one owner. | Planned |
| Scale Streaming Service aggressively to 8-12 replicas in full benchmark profiles and tune dummy segment payloads per profile. | Streaming service Compose scale profile, `STREAMING_SEGMENT_COUNT`, `STREAMING_SEGMENT_SIZE_BYTES`, Kafka producer config. | Streaming generates the highest request and event volume. | CPU, network egress, Kafka producer pressure. | Workload table: ~40,000 playback events/s; `REQUIREMENTS.md` M-04, M-05, M-06 require simulated streaming and playback events. | Keeps smoke/calibration cheap with smaller payloads while allowing full benchmark payload simulation. | k6 streaming scenario, Kafka producer latency, gateway response-size and latency metrics. | Planned |
| Tune Kafka topic partitions and producer batching for playback events before adding expensive broker capacity. | Kafka service config, kafka-init topic creation, Streaming producer config, Analytics/Recommendation consumer settings. | Playback events dominate the workload and consumer parallelism is limited by partitions. | Broker CPU/disk/network, consumer lag, producer latency. | Workload table: ~40,000 playback events/s; `TECH-STACK.md` selects Kafka; Streaming, Analytics, Recommendation, and Notification use Kafka. | Better throughput per broker before paying for more broker resources. | Kafka exporter metrics, consumer lag, k6 streaming throughput, broker disk usage. | Planned |
| Scale Analytics consumers to 3-6 only with matching Kafka partitions and ClickHouse insert capacity. | Analytics service scale profile, Kafka consumer group settings, ClickHouse insert/batching configuration. | Analytics must ingest playback history and serve chart/history queries; consumers beyond partition/DB capacity waste resources. | ClickHouse CPU/memory/disk, Kafka lag, app CPU. | `REQUIREMENTS.md` M-13, M-14; workload table playback event volume; `TECH-STACK.md` ClickHouse choice. | Avoids over-scaling consumers that create too many small ClickHouse inserts. | Kafka lag, ClickHouse insert latency, history/chart latency, parts count. | Planned |
| Keep Search app replicas at 4-6 but treat OpenSearch as the likely expensive bottleneck. | Search service scale profile, OpenSearch memory/CPU limits, index settings. | Search load is high but OpenSearch heap and query/index pressure usually dominate cost. | Memory-heavy OpenSearch heap, CPU, disk I/O. | Workload table: ~4,000 catalog/search requests/s; `REQUIREMENTS.md` M-12; `TECH-STACK.md` OpenSearch choice. | Prevents adding app replicas when the search engine is saturated. | k6 search scenario, OpenSearch heap/query latency/rejected task metrics. | Planned |
| Prevent duplicate Search index rebuilds during scaled runs by planning a single indexing-owner mode or preloaded index volume. | Search startup configuration, future Compose profile/env var, OpenSearch data volume. | Multiple Search replicas rebuilding the same index waste CPU, memory, and disk I/O. | OpenSearch indexing CPU, heap, disk writes, startup time. | Current Compose exposes `SEARCH_INDEXING_ENABLED` and `SEARCH_INDEX_BATCH_SIZE`; Search Service owns OpenSearch index data. | Faster and cheaper scaled startup. | Scaled startup logs, index document count, OpenSearch refresh/merge metrics. | Planned |
| Scale Playlist Service modestly to 2-3 replicas and focus on PostgreSQL indexes/connection limits. | Playlist service scale profile, playlist-db indexes, datasource pool config. | Playlist mutations are only ~200/s and are transactional, so too many replicas can increase DB contention. | DB locks, connection count, write latency. | Workload table: ~200 playlist mutations/s; `REQUIREMENTS.md` M-10, M-11. | Avoids spending on replicas that shift the bottleneck to PostgreSQL. | k6 playlist scenario, DB lock/connection metrics, mutation p95 latency. | Planned |
| Scale Recommendation Service to 3-5 replicas with Redis cache hit ratio as the gating metric. | Recommendation service scale profile, Redis TTL/maxmemory config, recommendation-db indexes. | Recommendation endpoints are read-heavy and can reuse cached daily mix/similar responses. | PostgreSQL query CPU, app CPU, Redis memory. | `REQUIREMENTS.md` M-15, M-16; current Compose includes Redis and configurable `RECOMMENDATION_CACHE_TTL`. | Redis can reduce repeated database work if hit ratio is healthy. | Redis hit ratio/memory/evictions, recommendation p95 latency, DB query metrics. | Planned |
| Keep Notification Service small at 1 replica normally and 2 only when Kafka lag or failure-isolation tests require it. | Notification service scale profile, MongoDB indexes, notification consumer metrics. | Notification is internal and not part of the high-rate public target metrics. | Avoids idle container CPU/memory and MongoDB pressure. | `ARCHITECTURE.md` says Notification is internal only; `REQUIREMENTS.md` M-17 requires event consumption and persistence but no public client API. | Reduces idle runtime cost. | Kafka lag for notification group, MongoDB insert latency, notification health. | Planned |
| Keep stateful infrastructure single-node for default/smoke/calibration and scale vertically first, except where benchmark evidence proves otherwise. | PostgreSQL containers, Redis, OpenSearch, ClickHouse, MongoDB, Kafka Compose profiles. | Stateful scaling in Docker Compose is storage- and configuration-sensitive; blind `--scale` is unsafe. | Memory, disk, data consistency, duplicated storage. | `REQUIREMENTS.md` M-26 requires dedicated persistence; `TECH-STACK.md` assigns specific persistence engines. | Avoids expensive idle clusters and data-management complexity in low-scale runs. | Per-profile Compose validation, database health checks, benchmark bottleneck metrics. | Planned |
| Make observability profile-aware while preserving required Prometheus/Grafana evidence. | Prometheus retention config, Grafana provisioning, exporters, k6 summaries. | Observability is required but time-series retention and exporters can add memory/disk cost. | Prometheus disk, memory, exporter containers. | `REQUIREMENTS.md` M-24 and S-03 require metrics and dashboard; M-21 requires load generation. | Lower smoke/calibration cost while keeping benchmark evidence. | Prometheus target checks, Grafana dashboard parse, k6 summary artifacts. | Planned |
| Use staged k6 test phases from smoke to isolated probes to calibration to 100k/1m profiles. | `load-generator/k6`, future scripts/profiles, README/test docs. | Full target load can saturate the host or k6 before service bottlenecks are understood. | Wasted compute during failed oversized runs; misleading benchmark evidence. | Workload table and existing k6 requirement M-21. | Finds bottlenecks earlier with less CPU/memory spend. | k6 dropped iterations, request failure rate, scenario throughput, generated summaries. | Planned |
