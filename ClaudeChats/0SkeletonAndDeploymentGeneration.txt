  ---
  Phase 0 — Implementation Plan: Shared Deployment Environment

  Assumptions (explicit)

  1. PostgreSQL isolation — Each service requiring PostgreSQL gets its own
  separate container (auth-db, catalog-db, playlist-db, recommendation-db). A
  single container with multiple databases would technically work but conflates
  service boundaries. Separate containers fully satisfies M-26.
  2. Kafka broker — A single Kafka broker with Zookeeper is used. KRaft mode is
  avoided because Zookeeper-based Kafka 3.x has broader ecosystem support and is
   well-tested in compose setups. A single broker is sufficient for the
  benchmarking workload.
  3. Load generator — k6 is chosen: it runs in Docker natively, is scriptable in
   JavaScript, and produces Prometheus-compatible metrics. No requirement
  specifies a tool; k6 satisfies M-21 and integrates with M-24.
  4. JWT key distribution — Auth Service owns the RSA key pair. The public key
  is written to a named Docker volume (jwt-keys) that all other services mount
  read-only. This avoids hardcoding the key and keeps the live key available at
  runtime without baking it into images.
  5. Grafana is included — S-03 is a should-have. Including it now costs nothing
   and avoids rework in Phase 10.
  6. Frontend port — 3000 (Vite dev default) served behind Nginx in the
  container. Grafana is mapped to host port 3001 to avoid collision.
  7. OpenSearch single node — discovery.type=single-node. Sufficient for
  non-production benchmarking.
  8. ClickHouse single node — No clustering. Sufficient for analytics
  aggregation at benchmarking scale.
  9. No API gateway — ARCHITECTURE.md defines no gateway. Each service is
  reachable from the frontend and load generator by service name within the
  Docker network, and by host port externally.
  10. Top-level .env — A single top-level .env supplies values consumed by
  docker-compose.yml (ports, versions, resource limits). Per-service
  .env.example files document service-specific variables and are not mounted by
  compose; services read their config from the environment: block in compose.

  All Spring Boot services use internal port 8080 (not 8080/tcp on the host to
  avoid conflicts). This is controlled by server.port=8080 in each
  application.yml.

  Resource limits (M-20)

  Each service block will include a deploy.resources.limits section with cpus
  and memory variables sourced from .env:

  deploy:
    resources:
      limits:
        cpus: "${AUTH_SERVICE_CPU_LIMIT:-1.0}"
        memory: "${AUTH_SERVICE_MEMORY_LIMIT:-512m}"

  Default values are set conservatively in .env.example. Operators change one
  variable to adjust a single service without editing the compose file.

  depends_on / health-check strategy

  Infrastructure containers will expose healthcheck blocks. Application services
   declare depends_on with condition: service_healthy against their required
  infrastructure. This ensures compose startup ordering without arbitrary sleep
  hacks.
