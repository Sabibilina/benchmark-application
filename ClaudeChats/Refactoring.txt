⏺ The plan is complete. Here is a summary of findings and decisions:

  ---
  Cost Efficiency Plan — Summary

  Main cost drivers identified

  CPU (highest impact):
  1. SecureRandom in streaming service — cryptographic entropy generation at 40K
   events/s peak. SecureRandom.nextBytes() is 30–100× slower than
  ThreadLocalRandom for the same task. Since segments are synthetic (requirement
   W-03), cryptographic quality is irrelevant.
  2. JVM heap under-sizing across all 8 services — bare java -jar app.jar with
  no flags causes Java 21 to allocate only 25% of the 512 MB container limit
  (128 MB heap). Spring Boot 3's baseline footprint alone is 150–200 MB, so
  every service runs in constant GC stress. Evidence: load test latency spikes
  under concurrent load despite services appearing healthy.
  3. No catalog page cache — 4,000 catalog/search req/s hitting PostgreSQL
  directly for queries that return identical results for the same genre+page
  combination.

  Memory:
  4. 64 KB segment payloads — At 20K concurrent users each mid-stream, this is
  ~1.28 GB of byte arrays in JVM heap simultaneously — alone exceeding the 512
  MB container limit before the JVM fix.

  Disk I/O:
  5. Kafka 24-hour log retention — At peak throughput, this accumulates hundreds
   of GB. Consumers drain the topic in seconds; 2 hours of retention provides a
  full restart-recovery buffer.
  6. Redis AOF persistence — unnecessary for a pure recommendation cache that is
   rebuilt from Kafka events on restart.
  7. Prometheus 15-day TSDB retention — 3 days covers all benchmark use cases.

  9 prioritised changes (in COST-AWARE-DECISIONS.md)
