  6. Seed Logic (DataSeeder)

  - Implements ApplicationRunner (runs after Spring context is fully up)
  - Guard: if (songRepository.count() > 0) return; — idempotent across restarts
  - Load ClassPathResource("data/songs.csv")
  - Parse with
  CSVFormat.DEFAULT.builder().setHeader().setSkipHeaderRecord(true).build()
  - Map each CSVRecord → Song entity; skip rows where title or artist is blank;
  log warning
  - Bulk-insert via JdbcTemplate.batchUpdate() in chunks of
  CATALOG_SEED_BATCH_SIZE (default 500)
  - Log one INFO line per batch and a final summary

  Why JdbcTemplate.batchUpdate() over repository.saveAll():
  saveAll() triggers one INSERT per entity through JPA's dirty-checking
  pipeline. For a dataset that may contain tens of thousands of rows,
  JdbcTemplate.batchUpdate() is significantly faster and avoids OOM from loading
   all entities into the first-level cache.

  7. JWT Validation

  The catalog service never calls Auth Service at runtime (satisfies M-03). It
  validates tokens locally using the RSA public key written by Auth Service to
  the shared jwt-keys Docker volume (D-005, D-027).

  JwtPublicKeyConfig: reads JWT_PUBLIC_KEY_PATH, decodes PEM → RSAPublicKey bean
   (same decode logic as JwtConfig.loadKeys() in Auth Service).

  JwtAuthenticationFilter:
  1. Read Authorization: Bearer <token> header; skip filter if absent or
  malformed (returns 401 downstream via SecurityConfig)
  2. Jwts.parser().verifyWith(rsaPublicKey).build().parseSignedClaims(token)
  3. Extract sub claim → set UsernamePasswordAuthenticationToken in
  SecurityContextHolder
  4. On JwtException or any parse error → clear context, write 401 Unauthorized

  JWT claims from Auth Service (confirmed from AuthService.java:68-73):
  - sub = user UUID (Long → String)
  - username = username string
  - Standard iat, exp
