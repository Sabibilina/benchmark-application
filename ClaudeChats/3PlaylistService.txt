  Endpoints and Behavior

  All endpoints require a valid JWT (Authorization: Bearer <token>). The sub
  claim is used as userId throughout.

  Method: GET
  Path: /playlists
  Body / Params: —
  Returns: 200 list of PlaylistSummaryResponse
  Notes: Lazily creates Liked Songs if absent; returns only authenticated user's

    playlists
  ────────────────────────────────────────
  Method: POST
  Path: /playlists
  Body / Params: CreatePlaylistRequest
  Returns: 201 PlaylistResponse
  Notes: Cannot create a playlist named Liked Songs (reserved)
  ────────────────────────────────────────
  Method: GET
  Path: /playlists/:id
  Body / Params: —
  Returns: 200 PlaylistResponse
  Notes: Returns 404 if not found or not owned by user
  ────────────────────────────────────────
  Method: PATCH
  Path: /playlists/:id
  Body / Params: UpdatePlaylistRequest
  Returns: 200 PlaylistResponse
  Notes: Only name is mutable; cannot rename Liked Songs
  ────────────────────────────────────────
  Method: DELETE
  Path: /playlists/:id
  Body / Params: —
  Returns: 204
  Notes: Returns 400 if isLikedSongs = true; 404 if not found or not owned
  ────────────────────────────────────────
  Method: POST
  Path: /playlists/:id/tracks
  Body / Params: AddTrackRequest
  Returns: 201 PlaylistResponse
  Notes: 409 if song already in playlist; 404 if playlist not found/not owned
  ────────────────────────────────────────
  Method: DELETE
  Path: /playlists/:id/tracks/:songId
  Body / Params: —
  Returns: 204
  Notes: 404 if track not in playlist or playlist not found/not owned
  ────────────────────────────────────────
  Method: PATCH
  Path: /playlists/:id/tracks/reorder
  Body / Params: ReorderTracksRequest
  Returns: 200 PlaylistResponse
  Notes: Validates all provided songIds belong to playlist; updates positions
    atomically in a @Transactional block

  Track position: 0-indexed integer. New tracks appended at max(position) + 1.
  Reorder replaces all positions from the provided ordered list.

  ---
  Persistence Approach

  Schema — V1 (playlists):
  id          UUID         PRIMARY KEY  DEFAULT gen_random_uuid()
  user_id     VARCHAR(255) NOT NULL
  name        VARCHAR(255) NOT NULL
  is_liked_songs  BOOLEAN  NOT NULL     DEFAULT false
  created_at  TIMESTAMPTZ  NOT NULL     DEFAULT now()
  updated_at  TIMESTAMPTZ  NOT NULL     DEFAULT now()

  INDEX ON (user_id)
  UNIQUE (user_id, name)   -- no two playlists with same name per user

  Schema — V2 (playlist_tracks):
  id          UUID         PRIMARY KEY  DEFAULT gen_random_uuid()
  playlist_id UUID         NOT NULL     REFERENCES playlists(id) ON DELETE
  CASCADE
  song_id     VARCHAR(255) NOT NULL
  position    INTEGER      NOT NULL
  added_at    TIMESTAMPTZ  NOT NULL     DEFAULT now()

  UNIQUE (playlist_id, song_id)    -- no duplicate tracks in a playlist
  INDEX ON (playlist_id, position)

  Flyway ddl-auto=validate. No shared database (M-26).

  ---
  Messaging Approach

  PlaylistEventPublisher sends JSON to topic playlist-events on Kafka
  (spring-kafka, JsonSerializer). Events are fire-and-forget (no await; failures
   logged, not propagated to client). Published on:
  - PLAYLIST_CREATED
  - PLAYLIST_UPDATED (name change or track modification)
  - PLAYLIST_DELETED

  Event payload:
  {
    "eventType": "TRACK_ADDED",
    "playlistId": "uuid",
    "userId": "username",
    "playlistName": "Workout Mix",
    "timestamp": "2026-05-13T10:00:00Z"
  }

  The Notification Service (Phase 8) consumes playlist-events to satisfy M-17.
  The topic name must be consistent; it is documented as the authoritative
  reference here.

  ---
  Missing Information (Not Invented)

  1. Track metadata in TrackResponse: Spec says "manages track operations" but
  does not specify whether track details beyond songId (title, artist, etc.)
  should be included in playlist responses. This service has no direct access to
   the Catalog DB. Decision needed: either return songId-only and let the
  frontend enrich from catalog, or add a GET /catalog/songs/:id call per track.
  The plan assumes songId-only (TrackResponse has songId, position, addedAt) to
  avoid coupling. If catalog enrichment is required, it must be a separate
  inter-service call with circuit breaker (M-23) — that detail is not in scope
  of this plan.
  2. Track count vs. full track list in GET /playlists: The summary list
  endpoint — unclear if it should return just count or full tracks. The plan
  returns PlaylistSummaryResponse (id, name, trackCount, isLikedSongs) for GET 
  /playlists and full PlaylistResponse (including tracks) for GET 
  /playlists/:id.
