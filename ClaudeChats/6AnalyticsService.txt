  ---
  Persistence Approach

  ClickHouse table (SchemaInitializer runs on startup):

  CREATE TABLE IF NOT EXISTS playback_events (
      event_type   String,
      user_id      String,
      song_id      String,
      occurred_at  DateTime64(3, 'UTC')
  ) ENGINE = MergeTree()
  ORDER BY (user_id, occurred_at);

  - No Flyway — ClickHouse dialect requires Flyway Teams (D-110).
  - IF NOT EXISTS makes the initialiser idempotent across restarts.
  - ORDER BY (user_id, occurred_at) makes history queries a primary-key range
  scan.
  - Global chart aggregation scans all rows grouped by song_id WHERE event_type 
  = 'play.started'.

  ---
  Messaging Approach

  - Consumer group: analytics-service
  - Topic: playback-events (D-077)
  - Deserialization: StringDeserializer (key + value); PlaybackEventConsumer
  decodes the JSON string via ObjectMapper into PlaybackEventRecord. This avoids
   dependence on __TypeId__ headers emitted by streaming-service's
  JsonSerializer (D-112).
  - Error handling: deserialization failures are logged and skipped; Kafka
  commit proceeds so the consumer doesn't stall.

  ---
  JWT Validation

  Same pattern as all prior consumer services (D-040, D-064, D-084, D-099):
  - JwtPublicKeyConfig — reads RSA public key from JWT_PUBLIC_KEY_PATH
  - JwtAuthenticationFilter — validates Bearer token on every request, sets
  SecurityContextHolder
  - SecurityConfig — permits /actuator/**, requires auth on all other paths
  - UserDetailsServiceAutoConfiguration excluded on @SpringBootApplication

  Fix 1 — ClickHouseTestContainer: missing credentials (D-129)

  ClickHouse 24.3's entrypoint explicitly disables network access when neither

  127.0.0.1 inside itself, so testcontainers' host-side port mapping could never
   reach the HTTP interface.

  Change: added three env vars to ClickHouseTestContainer:
  .withEnv("CLICKHOUSE_USER", "default")
  .withEnv("CLICKHOUSE_PASSWORD", "")
  .withEnv("CLICKHOUSE_DEFAULT_ACCESS_MANAGEMENT", "1")

  Fix 2 — ClickHouseTestContainer: wrong wait strategy (D-130)

  ClickHouse writes all server logs to files (/var/log/clickhouse-server/), not
  stdout. LogMessageWaitStrategy watches stdout, so it never saw the "Ready for
  connections" message and timed out after 120 s. Switched to HTTP /ping (the
  same health signal used by the compose healthcheck) with an explicit 180 s
  timeout to accommodate slower Docker-in-Docker startup.

  Change: wait strategy replaced:
  Wait.forHttp("/ping")
      .forPort(8123)
      .forStatusCode(200)
      .withStartupTimeout(Duration.ofSeconds(180))
