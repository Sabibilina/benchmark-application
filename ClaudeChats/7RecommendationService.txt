  ---
  Persistence and Messaging

  PostgreSQL — Flyway schema (ddl-auto=validate):

  V1__create_recommendation_songs.sql — recommendation_songs(id SERIAL PK, 
  track_id VARCHAR UNIQUE, title VARCHAR, artist VARCHAR, genre VARCHAR, tempo 
  DOUBLE PRECISION, year INT) + indexes on (genre), (genre, tempo).

  V2__create_play_events.sql — play_events(id BIGSERIAL PK, user_id VARCHAR NOT 
  NULL, song_id BIGINT NOT NULL, event_type VARCHAR NOT NULL, occurred_at 
  TIMESTAMP NOT NULL DEFAULT now()) + indexes on (user_id, occurred_at),
  (song_id).

  Seeding: RecommendationSeeder (ApplicationRunner, order=1) loads songs.csv via
   JdbcTemplate.batchUpdate() in batches of 500. Skips if
  count(recommendation_songs) > 0. Inserts only: track_id, title (track_name),
  artist (artist_name), genre, tempo, year (extracted from release_date).

  Kafka consumer: @KafkaListener on playback-events topic with group
  recommendation-service. Deserialises raw JSON string to PlaybackEventRecord
  via ObjectMapper. Persists only events where eventType = "play.started" to
  play_events. Other event types are acknowledged and dropped.

  Redis cache:
  - RedisConfig defines a RedisTemplate<String, String> with Jackson JSON
  serialization.
  - RecommendationService reads/writes cache explicitly (not @Cacheable) for
  control over TTL.
  - Cache keys: daily-mix:{userId} and similar:{songId}.
  - Cache failures are caught and logged; the service falls back to PostgreSQL
  queries.

  Recommendation logic:

  Daily Mix:
  1. Check Redis cache for daily-mix:{userId}.
  2. Query play_events for the user's most-played song IDs (last 30 days,
  play.started, ORDER BY count DESC, LIMIT 10).
  3. Look up genres of those songs in recommendation_songs → find the most
  frequent genre.
  4. Sample RECOMMENDATION_DAILY_MIX_SIZE songs from that genre (excluding
  already-played IDs) using recommendation_songs.
  5. Fallback chain: (a) if no user history → take the globally top-played song
  IDs across all users and repeat; (b) if play_events is empty → return
  RECOMMENDATION_DAILY_MIX_SIZE random songs from recommendation_songs.
  6. Write result to Redis with TTL. Return response.

  Similar Songs:
  1. Check Redis cache for similar:{songId}.
  2. Parse songId as Long. Look up recommendation_songs by id = songId.
  3. If found: query songs with genre = target.genre AND ABS(tempo - 
  target.tempo) <= 20 AND id != songId LIMIT RECOMMENDATION_SIMILAR_SIZE.
  4. If fewer than RECOMMENDATION_SIMILAR_SIZE results, fill with random
  same-genre songs.
  5. If songId not found: return RECOMMENDATION_SIMILAR_SIZE random songs from
  the full catalog.
  6. Write result to Redis with TTL. Return response.

  Explicit Assumptions

  1. ID alignment: recommendation_songs.id (SERIAL starting at 1) will match
  catalog_songs.id for the same song because both services load the same CSV
  rows sequentially. If IDs diverge (e.g., one service loads a different CSV
  version), similar/:songId falls back gracefully to returning random songs
  rather than failing.
  2. Event format: play.started events emitted by the Streaming Service contain
  at minimum userId (string), songId (serialised as string or number), and
  eventType (string). The PlaybackEventRecord DTO uses Long songId and the
  ObjectMapper handles numeric/string coercion.
  3. Redis availability: Redis is always available in compose (healthcheck
  exists). In integration tests, Redis is replaced by RedisTestContainer. If
  Redis is down at runtime, recommendations are still served from PostgreSQL
  (cache miss is not an error).
  4. Missing detail — songs.csv path: The CSV file must be physically present at
   src/main/resources/data/songs.csv. Since the same file is already in
  services/catalog-service/src/main/resources/data/songs.csv, it will be copied
  there during code generation. There is no separate download step.
