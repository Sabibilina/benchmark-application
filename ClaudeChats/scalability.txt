  (BCrypt is expensive), consumers capped at Kafka partition count.

  indexes for all four PostgreSQL DBs, OpenSearch shard/heap sizing, ClickHouse
  batch-insert requirement, Redis maxmemory + LRU policy.
  - §10 — The single most critical prerequisite: pre-creating playback-events

  replica you add is idle.
  - §12 — Resilience4j circuit breakers, nginx rate limiting, ClickHouse
  dead-letter pattern.
  - §13 — Four Prometheus/Grafana panel groups with exact metric names and alert
   thresholds.
  - §14 — Six k6 phases from 5 VU smoke to 10 K VU peak, with explicit "do not
  skip" ordering.
  - §15 — A 15-step sequenced scaling order so low-impact services are not
  optimized before high-impact prerequisites.
  - §16 — Seven concrete risks (JWT race, Kafka partition immutability, nginx
  DNS caching, host RAM budget, ClickHouse part explosion, etc.).

⏺ Everything is in place. Here is a precise account of what was implemented and

  ---
  What changed and why

  1. init-kafka one-shot container

  File: docker-compose.yml

  Creates playback-events (12 partitions) and playlist-events (6 partitions)
  before any service starts. KAFKA_AUTO_CREATE_TOPICS_ENABLE=false was added to
  the Kafka broker so services cannot accidentally trigger 1-partition topic
  creation by connecting first. All Kafka-producing/consuming services declare
  init-kafka: condition: service_completed_successfully in depends_on. Partition
   counts are immutable after creation — this must be done before first boot.

  2. nginx-lb dedicated load balancer

  Files: infra/nginx-lb/nginx.conf, docker-compose.yml

  Uses resolver 127.0.0.11 valid=5s with set $upstream_<svc> 
  "service-name:8080"; proxy_pass http://$upstream_<svc> on every location. This
   is the only pattern in open-source nginx that causes per-request DNS
  re-resolution — without the variable, nginx resolves once at startup and all
  traffic goes to one replica forever. Rate limiting: auth at 20 r/s (BCrypt is
  CPU-expensive), streaming at 200 r/s, general API at 500 r/s. Streaming has
  proxy_buffering off so 64 KB segment payloads pass through without buffering.

  3. Frontend nginx routed through nginx-lb

  File: frontend/nginx.conf

  All /api/* locations now proxy_pass http://nginx-lb/<path> instead of directly
   to services. The path rewriting pattern is preserved identically — only the
  target changes from individual service names to nginx-lb. This eliminates the
  single-IP DNS cache problem in the SPA nginx while also consolidating all
  traffic through one observable, rate-limited point.

  4. container_name removed from 7 application services

  File: docker-compose.yml

  auth-service retains its container_name because it must stay at 1 replica on a
   fresh volume. The JwtConfig.java key generation is not atomic — two
  simultaneous instances seeing absent files both generate different key pairs,
  and whichever writes last determines what other services read from the volume.
   All other services have no container_name, enabling docker compose up --scale
   <service>=N without port or name conflicts.

  5. auth-service: service_healthy in all jwt-keys consumers

  File: docker-compose.yml

  Every service that mounts jwt-keys:/jwt-keys:ro now depends on auth-service
  being healthy. This adds ~30–60 s startup latency but eliminates the crash
  race where a service starts its @PostConstruct key-loader before the RSA files
   exist.

  6. ClickHouse batch insert

  Files: BatchEventBuffer.java (new), PlaybackEventConsumer.java,
  AnalyticsRepository.java, AnalyticsService.java,
  AnalyticsServiceApplication.java, application.yml

  BatchEventBuffer accumulates events in a synchronized ArrayList and flushes
  via JdbcTemplate.batchUpdate() in batches of 500 or every 5 seconds. The
  existing recordEvent() API is unchanged (tests still pass). The new
  recordEvents(List<>) path is only called from the buffer. @EnableScheduling
  added to the application class to activate the @Scheduled flush.

  7. Resource limits, PostgreSQL tuning, Redis and OpenSearch tuning

  File: docker-compose.yml, .env.example

  All infrastructure containers now have explicit CPU and memory limits
  (previously none). Redis gets --maxmemory 512mb --maxmemory-policy allkeys-lru
   --appendfsync everysec. OpenSearch heap goes from 512 m to 1 g. PostgreSQL
  gets per-instance -c shared_buffers / work_mem / max_connections tuned to its
  workload (catalog-db gets more because it serves the largest read dataset).

  8. Prometheus exporters + DNS-SD for scaled services

  Files: docker-compose.yml, infra/prometheus/prometheus.yml

  Added postgres-exporter ×4, redis-exporter, kafka-exporter. Scaled services
  (all except auth) use dns_sd_configs so Prometheus discovers and scrapes every
   replica independently — without this, it only scrapes whichever IP Docker DNS
   returns on the first query.

  9. Grafana scaling dashboard

  File: infra/grafana/dashboards/scaling.json

  19 panels across 6 rows: request rate, error rate, latency, JVM heap, CPU,
  HikariCP saturation, PostgreSQL connections and sequential scans, Kafka
  consumer lag (per group), Redis hit rate and memory, OpenSearch heap.
  Threshold lines are drawn at the alert values defined in SCALABILITY.md §13.

  10. k6 full user journey with 6 scenarios

  File: load-generator/scripts/main.js

  Implements the complete flow: register → login → browse catalog → search →
  stream manifest → stream segment → complete event (→ Kafka) → playlist →
  recommendations → analytics → notifications. Each VU registers once and reuses
   the token. Six selectable scenarios via K6_SCENARIO: smoke, catalog_stream,
  kafka_pipeline, ramp, stress, soak. All traffic routes through
  NGINX_LB_URL=http://nginx-lb.
