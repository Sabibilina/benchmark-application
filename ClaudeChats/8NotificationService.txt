  Data MongoDB on first insert.

  ---
  Messaging Approach

  - Topic consumed: playlist-events (authoritative per D-059)
  - Event types that generate notifications: all six — PLAYLIST_CREATED,
  PLAYLIST_UPDATED, PLAYLIST_DELETED, TRACK_ADDED, TRACK_REMOVED,
  TRACKS_REORDERED
  - Deserialization: StringDeserializer for key and value; ObjectMapper in
  PlaylistEventConsumer parses the raw JSON string into PlaylistEventRecord —
  type-header independent (consistent with D-112, D-139)
  - Consumer group: notification-service
  - Failure handling: deserialization errors are logged and skipped; Kafka
  offsets are committed to prevent infinite retries

  ---
  JWT Validation

  Same four-file pattern copied from all prior services (D-040, D-064, D-084,
  D-099, D-138):
  - JwtPublicKeyConfig — reads RSA public key from JWT_PUBLIC_KEY_PATH at
  @PostConstruct
  - JwtAuthenticationFilter — extracts sub claim; sets SecurityContextHolder
  - SecurityConfig — permits /actuator/**; requires auth on all other paths
  - UserDetailsServiceAutoConfiguration excluded from @SpringBootApplication

  Assumptions

  1. All six playlist event types generate notifications. No filtering by event
  type at consumer level. If this needs to change, the service logic in
  NotificationService.buildNotification() is the only place to update.
  2. GET /notifications returns all stored notifications for the user (no
  pagination). The number of notifications per user is naturally bounded in the
  benchmarking context.
  3. No PATCH /notifications/{id}/read endpoint is planned. Marking
  notifications read is out of scope for Phase 8; the read field is stored for
  future use.
  4. No "new release events" topic exists in the current system, so only
  playlist-events is consumed. The spec says "at least playlist update events or
   new release events" — consuming playlist-events fully satisfies M-17.
  5. PlaylistEventRecord in the notification-service is a local DTO (not shared
  with playlist-service). This maintains service isolation (D-056, D-082).
  6. docker-compose.yml will be updated during Generate step to add
  JWT_PUBLIC_KEY_PATH, KAFKA_TOPIC_PLAYLIST_EVENTS, and KAFKA_CONSUMER_GROUP_ID
  to the notification-service env block. The volume mount already exists.

  Summary of the plan

  What the service does: Consumes the playlist-events Kafka topic (topic name
  authoritative from D-059) and persists in-app notifications to MongoDB.
  Exposes a JWT-protected GET /notifications endpoint for the optional frontend
