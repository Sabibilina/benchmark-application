  ---
  Endpoints

  POST /auth/register — public
  - Request: { "username": "string", "email": "string", "password": "string" }
  - Validation: username non-blank ≤50 chars; email valid format; password ≥8
  chars
  - Success 201: { "token": "<RS256 JWT>" }
  - Conflict 409: username or email already exists
  - Error 400: validation failure (field-level detail in body)

  POST /auth/login — public
  - Request: { "username": "string", "password": "string" }
  - Success 200: { "token": "<RS256 JWT>" }
  - Error 401: unknown username or wrong password (undifferentiated, by design)

  ---
  JWT Details

  - Algorithm: RS256 (RSASSA-PKCS1-v1_5 + SHA-256)
  - Key size: 2048-bit RSA
  - Claims: sub (userId UUID as string), username, iat, exp
  - Key lifecycle: JwtConfig runs at startup; if /jwt-keys/private.pem is absent
   → generate fresh pair → write private.pem and public.pem; if present → load
  from disk. This is the authority write for the shared jwt-keys Docker volume
  (D-005). Other services read only public.pem.
  - Token TTL controlled by JWT_EXPIRATION_MS (default 3 600 000 ms = 1 h)

  ---
  Persistence

  Schema (Flyway V1__create_users_table.sql):
  CREATE TABLE users (
      id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
      username    VARCHAR(50)  UNIQUE NOT NULL,
      email       VARCHAR(255) UNIQUE NOT NULL,
      password_hash VARCHAR(255) NOT NULL,
      created_at  TIMESTAMPTZ DEFAULT now()
  );

  JPA: User entity mapped to users table. UserRepository extends
  JpaRepository<User, UUID> with findByUsername, existsByUsername,
  existsByEmail.

  DDL strategy: Flyway manages schema (not JPA ddl-auto);
  spring.jpa.hibernate.ddl-auto=validate so Hibernate confirms the schema
  matches entities without modifying it.

  Security Configuration

  - Spring Security SecurityFilterChain:
    - POST /auth/register and POST /auth/login — permitAll()
    - /actuator/** — permitAll() (required for compose healthcheck)
    - All other paths — authenticated() (auth-service has no other public
  endpoints, so this is a safe default)
  - Session management: STATELESS
  - CSRF: disabled
  - No http.formLogin() or http.httpBasic() — REST only

  ---
  Validation Steps

  1. mvn verify — all unit and integration tests pass locally.
  2. docker compose up auth-db auth-service — both containers reach healthy.
  3. curl -s http://localhost:8081/actuator/health → {"status":"UP"}.
  4. POST /auth/register with valid body → HTTP 201, response body contains
  token.
  5. Decode the token header → "alg":"RS256".
  6. POST /auth/login with the same credentials → HTTP 200, token present.
  7. POST /auth/login with wrong password → HTTP 401.
  8. POST /auth/register again with same username → HTTP 409.
  9. ls inside the jwt-keys volume (or via another container) → private.pem and
  public.pem exist.
  10. curl http://localhost:8081/actuator/prometheus → returns Prometheus
  metrics.

  ---
  Assumptions

  #: A-1
  Assumption: Login identifies the user by username, not email
  Reason: ARCHITECTURE.md does not specify the login identifier; username
  matches
    the registration field and keeps the schema simple.
  ────────────────────────────────────────
  #: A-2
  Assumption: POST /auth/register returns a JWT immediately (auto-login)
  Reason: The spec says "issues JWT access tokens" but does not restrict when;
    issuing on register saves a round-trip for clients and is the expected UX
  for
     REST auth.
  ────────────────────────────────────────
  #: A-3
  Assumption: Password minimum length is 8 characters
  Reason: Not stated in the spec; 8 chars is the minimum acceptable security
    practice.
  ────────────────────────────────────────
  #: A-4
  Assumption: JWT sub claim is the user's UUID
  Reason: Standard JWT practice; downstream services need a stable user
    identifier.
  ────────────────────────────────────────
  #: A-5
  Assumption: Flyway manages DDL; JPA ddl-auto=validate only
  Reason: Flyway migrations survive restarts correctly and are explict;
    create-drop or update would risk data loss on restart in a benchmarking
    context.
  ────────────────────────────────────────
  #: A-6
  Assumption: RSA key generation happens at Spring context startup inside
    JwtConfig
  Reason: The alternative (an entrypoint shell script) was considered but
    in-process generation keeps all logic in Java and avoids shell dependency on

    the JRE-only image.
  ────────────────────────────────────────
  #: A-7
  Assumption: /actuator/** is publicly accessible
  Reason: Required for compose healthcheck; consistent with D-007.

  ---
  Missing Details (not inventing — listed for review)

  - The spec does not define the exact shape of error response bodies (e.g., { 
  "error": "..." } vs RFC 7807 Problem Details). The plan uses a simple { 
  "message": "..." } envelope; this should be confirmed before code generation
  to ensure consistency across services.
  - The spec does not state whether the register response should include user
  metadata (id, username) alongside the token, or just the token. Plan assumes
  token-only response.
