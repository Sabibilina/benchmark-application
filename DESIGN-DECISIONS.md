## Phase S1 — Scaling Implementation Decisions (2026-05-21)

| ID | Decision | Why | Justified by | Affects |
|----|----------|-----|--------------|---------|
| D-S01 | `init-kafka` one-shot compose service creates `playback-events` (12 partitions) and `playlist-events` (6 partitions) before any application service starts. `KAFKA_AUTO_CREATE_TOPICS_ENABLE=false` on the Kafka broker. | Auto-creation with default partition count (1) is a silent correctness bug: Kafka consumers in the same group only process in parallel up to the partition count. One partition = one active consumer regardless of replica count. Disabling auto-creation and managing topics explicitly removes the ambiguity. 12 partitions for playback-events supports 4 analytics replicas + 3 recommendation replicas simultaneously (each group gets the full 12). | SCALABILITY.md §10; F-S01 | `docker-compose.yml` |
| D-S02 | Dedicated `nginx-lb` container (port 80) using `resolver 127.0.0.11 valid=5s` + `set $upstream_<svc> "<svc>:8080"; proxy_pass http://$upstream_<svc>;` pattern. | Standard nginx `proxy_pass http://service-name:port` resolves the hostname once at config load and caches the single IP. With `deploy.replicas > 1`, this routes all traffic to one replica. The variable-in-proxy_pass pattern forces per-request DNS resolution (within the 5-second TTL), distributing load across all replicas as Docker DNS returns them in round-robin order. | SCALABILITY.md §2; nginx resolver docs; F-S02 | `infra/nginx-lb/nginx.conf`, `docker-compose.yml` |
| D-S03 | Frontend nginx routes all `/api/*` to nginx-lb (`proxy_pass http://nginx-lb/<path>`) instead of directly to each service. | Frontend nginx had the same single-IP DNS caching issue as any nginx with static proxy_pass. Routing through nginx-lb consolidates rate limiting, access logging, and dynamic resolution at one layer. | SCALABILITY.md §2; F-S02 | `frontend/nginx.conf` |
| D-S04 | `container_name` removed from all 8 application services except `auth-service`. Auth keeps its name (and stays at `replicas: 1`) because the `JwtConfig.java` key generation on a fresh volume is not atomic — two simultaneous instances can each generate different key pairs and write them to the same files. | The fix for the race would be a separate `init-jwt-keys` one-shot container. That requires code changes to make auth-service load-only (never generate). Acceptable for now: auth-service BCrypt hashing is CPU-expensive enough that 1–2 instances handle the auth load of a benchmark run; scaling is deferred until the volume has been populated by a first clean boot. | SCALABILITY.md §16 Risk 1; F-S04; JwtConfig.java L45-51 | `docker-compose.yml` |
| D-S05 | All services that mount `jwt-keys:/jwt-keys:ro` now declare `auth-service: condition: service_healthy` in `depends_on`. | Without this dependency, catalog-service could start before auth-service generates the RSA keys. The key loader in each service uses `@PostConstruct` which throws IOException if the file is absent, crashing the service before it reaches the health check. Adding the dependency adds startup latency (~30–60 s) but eliminates the crash race on every fresh `docker compose up`. | JwtConfig.java; F-S04 | `docker-compose.yml` |
| D-S06 | `BatchEventBuffer` (new component in analytics-service) accumulates Kafka-consumed events and flushes them via `JdbcTemplate.batchUpdate()` in batches of 500 or every 5 s. `@EnableScheduling` added to `AnalyticsServiceApplication`. | ClickHouse's MergeTree creates one on-disk *part* per INSERT statement. At the event rates expected for 1 M users (~40 K events/s peak), individual inserts create ~40 K parts/s. ClickHouse recommends < 300 parts/s. Batching 500 events reduces parts/s by 1000×, eliminates "Too many parts" rejections, and improves merge throughput. The 5-second timer bounds delivery latency for low-traffic periods. | SCALABILITY.md §7; ClickHouse MergeTree docs; F-S03 | `BatchEventBuffer.java`, `PlaybackEventConsumer.java`, `AnalyticsRepository.java`, `AnalyticsService.java`, `AnalyticsServiceApplication.java` |
| D-S07 | `PlaybackEventConsumer` now injects `BatchEventBuffer` instead of `AnalyticsService` directly. The original `AnalyticsService.recordEvent()` is preserved unchanged for unit tests. A new `AnalyticsService.recordEvents(List<>)` delegates to `AnalyticsRepository.insertBatch()`. | Maintaining the existing `recordEvent()` signature means all existing unit and integration tests compile and pass without modification. The batch path is a parallel addition, not a replacement of the existing API. This follows an additive change strategy consistent with D-112 (analytics service original implementation decisions). | D-S06; backwards-compat of existing tests | `AnalyticsService.java`, `AnalyticsRepository.java` |
| D-S08 | Redis configured with `--maxmemory 512mb --maxmemory-policy allkeys-lru --appendfsync everysec`. `--save ""` disables RDB snapshots (AOF only). | allkeys-lru evicts the least-recently-used key when maxmemory is reached; this matches the recommendation cache TTL pattern (active user keys stay hot, expired keys are evicted first under memory pressure). `appendfsync everysec` reduces per-write disk latency from 1-2 ms (always) to near-zero (async) at the cost of up to 1 second of cache state on crash — acceptable for a non-durable cache. Disabling RDB snapshots avoids fork-on-save latency spikes during load tests. | SCALABILITY.md §8 | `docker-compose.yml` |
| D-S09 | OpenSearch heap increased from `-Xms512m -Xmx512m` to `-Xms1g -Xmx1g` (configurable via `OPENSEARCH_JAVA_OPTS`). | At 512 MB, OpenSearch hits GC pressure under query concurrency with a full songs index (~500 MB data). At 1 g, it has room for the field-data cache and segment buffers needed for concurrent search requests from 2–4 search-service replicas. The 31 GB compressed-OOP threshold is not approached. | SCALABILITY.md §6 | `docker-compose.yml`, `.env.example` |
| D-S10 | Prometheus uses `dns_sd_configs` for all scaled services (catalog, streaming, playlist, search, analytics, recommendation, notification) so that each replica is scraped independently. `auth-service` uses `static_configs` (single instance). | With `static_configs: targets: [streaming-service:8080]`, Prometheus scrapes one IP per scrape cycle (whichever Docker DNS returns first). With `dns_sd_configs`, Prometheus resolves the hostname, discovers all A records (one per replica), and scrapes each independently. This is essential for per-replica metrics visibility — without it, Grafana shows aggregate data from only one replica even when three are running. | SCALABILITY.md §13; Prometheus DNS-SD docs | `infra/prometheus/prometheus.yml` |
| D-S11 | Six k6 scenarios implemented in `main.js`, selectable via `K6_SCENARIO` env var. All scenarios route traffic through `NGINX_LB_URL=http://nginx-lb`. Phase ordering in scenarios (smoke → catalog_stream → kafka_pipeline → ramp → stress → soak) mirrors the SCALABILITY.md §14 test plan, ensuring each phase can serve as a stopping point for intermediate analysis. | SCALABILITY.md §14; benchmark design principles | `load-generator/scripts/main.js`, `docker-compose.yml` |
| D-S12 | PostgreSQL instances tuned via `-c` args in `command:` rather than a mounted `postgresql.conf`. `shared_buffers=512MB`, `work_mem=32MB` on catalog-db (largest read dataset); 256MB/16MB on others. `max_connections=200` on all instances (was default 100). | Inline `-c` args require no mounted config file and are version-agnostic. `shared_buffers` ≈ 25% of allocated container RAM. `work_mem` controls per-sort/hash RAM; 32 MB allows ORDER BY queries on the songs table without spilling to disk. Raising max_connections to 200 gives headroom for `4 replicas × 20 pool connections + exporters + DBA connections` without exhausting the limit. | SCALABILITY.md §5; PostgreSQL docs | `docker-compose.yml` |
